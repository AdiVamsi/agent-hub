# repo-pilot

> Karpathy's AutoResearch, but for repo maintenance. Point it at your issue backlog, wake up to a triaged inbox.

## What is this?

`repo-pilot` is an autonomous agent that optimizes open-source issue triage. It automatically labels, prioritizes, and deduplicates issues by tuning a simple, interpretable triage system.

The agent runs experiments on your issue backlog and learns what rules work best — maximizing the number of correctly triaged issues.

## The Pitch

> I pointed an AI agent at my repo's issue backlog. It ran 60 experiments overnight. My triage bot now correctly labels 89% of issues, up from 34%.

## How It Works

1. **Generate a test set:** 200 synthetic issues for a web framework (but works with real issues too)
2. **Start with a baseline:** Dead-simple triage rules (everything gets "needs-triage")
3. **Run experiments:** The agent tweaks `triage.py` to improve the `issues_resolved` metric
4. **Get results:** After optimization, the agent has learned what rules maximize correct triaging

## The Metric

**`issues_resolved`** = correct_labels*2 + correct_priorities*2 + correct_duplicates*3 - wrong_labels - wrong_priorities - wrong_duplicates*3

Higher is better. Rewards correct predictions, penalizes mistakes.

## Example Results

```
Baseline:      issues_resolved=-200 (label_acc=0.00, priority_acc=0.50, dup_f1=0.87)

```

## Quick Start

### 1. Generate Issues
```bash
python prepare.py generate
```
Creates `issue_backlog.json` (200 synthetic issues).

### 2. Run Baseline
```bash
python harness.py baseline
```
Shows how bad the default triage rules are.

### 3. Run Evaluation
```bash
python harness.py evaluate
```
Evaluates current `triage.py` against ground truth.

## The One Editable File

**`triage.py`** — This is what the agent modifies to improve the score.

It defines:
- `get_label_rules()` — Keyword patterns for auto-labeling
- `get_priority_rules()` — Rules for prioritization
- `get_duplicate_rules()` — Duplicate detection settings
- `classify_issue(title, body)` — Custom classification logic

Start with minimal/conservative rules. The agent experiments with adding and tuning rules.

## Key Optimization Targets

### Label Rules
- **bug**: crash, error, broken, fail, exception
- **feature**: add, support, implement, request, enable
- **docs**: documentation, typo, example, readme
- **performance**: slow, memory, lag, optimize
- **security**: vulnerability, xss, injection, auth, token
- **question**: how, help, usage, best practice
- **breaking-change**: breaking, upgrade, migration

### Priority Rules
- security issues → critical
- bugs → high/critical
- features → medium/high
- docs/questions → low

### Duplicate Detection
- Lower similarity threshold (0.95 is too conservative)
- Analyze both title and body
- Define synonym groups

## Example Experiment

```python
# BEFORE: get_label_rules() returns []
# Everything gets "needs-triage" label → lots of wrong predictions

# AFTER: add bug detection rules
def get_label_rules() -> list[dict]:
    return [
        {
            "label": "bug",
            "keywords": ["crash", "error", "broken", "fail", "exception"],
            "exclude_keywords": [],
            "priority_boost": 1,
        },
        # ... more rules
    ]

# Now bugs are correctly labeled → issues_resolved increases
```

## Files

- **`triage.py`** — The ONE editable file. Defines triage rules.
- **`harness.py`** — Evaluation harness. Fixed. Compares predictions to ground truth.
- **`prepare.py`** — Generates synthetic issue backlog. Run once.
- **`issue_backlog.json`** — Ground truth issues. Generated by prepare.py.
- **`program.md`** — Detailed optimization guide. Karpathy-style instructions.
- **`README.md`** — This file.
- **`pyproject.toml`** — Python metadata.

## Metrics

The harness reports:
- **issues_resolved** — Overall score. Target: > 200
- **label_accuracy** — % labels correct. Target: > 0.80
- **priority_accuracy** — % priorities correct. Target: > 0.65
- **duplicate_f1** — % duplicates correct. Target: > 0.70

## Implementation Notes

- Pure Python. No external dependencies.
- Deterministic issue generation (seed=42).
- 200 test issues. Realistic distribution (35% bugs, 25% features, 10% docs, etc.).
- 15% of issues are intentional duplicates.
- Scoring penalizes both false positives and false negatives equally.
