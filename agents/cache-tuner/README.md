# Cache Tuner — Optimize Eviction Policies for Maximum Hit Rate

Cache Tuner is an AutoResearch agent that optimizes cache eviction policies to maximize cache hit rate on realistic access patterns.

## Quick Start

```bash
# Generate synthetic access trace (5000 requests, 200 keys, tight constraints)
python prepare.py generate

# Run baseline FIFO eviction
python harness.py baseline

# Evaluate your policy against baseline
python harness.py evaluate
```

## The Problem

Your cache can't hold all data:
- **Max capacity:** 500 KB (typical cache size)
- **Max items:** 100 (limited eviction slots)
- **Access pattern:** 5000 requests over 200 unique keys
- **Current eviction:** FIFO (oldest inserted first)
- **Current hit rate:** ~30-40%

**Question:** How can you choose which items to evict to maximize hit rate?

## The Solution Space

We provide 8+ eviction hypotheses:

1. **LRU (Least Recently Used)** — Evict by `last_access` timestamp
2. **LFU (Least Frequently Used)** — Evict by `access_count`
3. **Size-Aware** — Evict largest items first
4. **Category-Aware** — Prioritize important data (user profiles > search results)
5. **ARC (Adaptive Replacement Cache)** — Track recent/frequent and recently evicted
6. **Scan Resistance** — Detect sequential patterns, don't cache them
7. **LRFU (Frequency + Recency)** — Weighted combination score
8. **Size-Weighted Frequency** — Maximize hits per byte

See `program.md` for detailed hypotheses and experimental plan.

## Evaluation

The harness replays the 5000-request trace through your policy:
- **Hit:** item was in cache → +1
- **Miss:** item not in cache → +0
- **Eviction:** cache full → call `on_evict` to make space

**Metric:** `hit_rate = hits / (hits + misses)`

Example output:
```
RESULT: hit_rate=0.7234 improvement_pct=95.5
```

## Anti-Gaming Safeguards

The harness enforces fairness:

1. **No direct access to trace:** `on_evict` cannot read `access_trace.json`
2. **No cheating on capacity:** Cache cannot exceed `max_capacity_bytes` or `max_items`
3. **Valid evictions:** `on_evict` must return a key actually in `cache_state`
4. **Proper returns:** `on_access` must return `bool`, `on_evict` must return `str`

## Hints

- Start with **LFU** (least frequently used) — simple, aligns with Zipfian distribution
- Progress to **LRU + category scoring** — balance frequency and value
- Advanced: **ARC** or **LRFU** — handle temporal patterns and bursts
- Don't forget: **size-aware** trade-offs and **scan resistance**

## Metrics to Track

- `hit_rate` — percentage of requests served from cache (MAIN METRIC)
- `hits` / `misses` — absolute counts
- `evictions` — how many times you had to make room
- `cache_bytes` / `cache_items` — cache fullness at end

Good policies should:
- ✓ Hit rate ≥ 0.70 (2x baseline)
- ✓ Few evictions per 100 requests
- ✓ Use cache space efficiently
- ✓ Adapt to Zipfian and temporal patterns
